Hermes/리오로 매일 블로그를 쓰고, 리포트를 만들고, 파일을 정리하고, 빌드까지 자동화하다 보면 가장 유혹적인 설계는 “가능하면 끝까지 자동으로 실행하게 만들자”입니다. 하지만 에이전트 자동화가 강력해질수록 더 중요한 것은 완전 자동화가 아니라 **실행 전 검문소**입니다. 오늘 AI/ML 트렌드에서 Mistral Vibe 같은 장기 실행 에이전트와 LLM을 활용한 공격 사례가 동시에 보인 것도 같은 메시지를 줍니다. 에이전트가 일을 잘하게 되는 만큼, 어디서 멈추고 확인해야 하는지도 명확해야 합니다.

Hermes/리오 작업을 설계할 때는 먼저 행동을 세 등급으로 나누는 편이 좋습니다. 첫째, 읽기와 분석입니다. 파일을 읽고, 최근 글을 확인하고, 보고서를 요약하고, 후보를 고르는 일은 비교적 안전합니다. 둘째, 내부 쓰기입니다. 초안 작성, 로컬 파일 생성, 상태 파일 업데이트처럼 되돌릴 수 있는 작업은 자동화해도 됩니다. 셋째, 외부 부작용입니다. 이메일 발송, 공개 게시, git push, 배포, 서비스 재시작, 결제, 주문 같은 일은 별도 검문소가 필요합니다. 이 구분이 없으면 에이전트는 “편리한 비서”에서 “왜 그런 행동을 했는지 추적하기 어려운 프로세스”가 됩니다.

블로그 크론을 예로 들면 검문소는 생각보다 구체적입니다. 글을 쓰기 전에는 오늘의 입력과 최근 글을 비교해 중복 주제를 피해야 합니다. 파일을 저장한 뒤에는 front matter, 카테고리, 태그, 민감정보 노출 여부를 확인해야 합니다. 커밋 전에는 `git status`로 의도하지 않은 변경이 섞였는지 봐야 합니다. 빌드 후에는 생성된 HTML이 실제로 만들어졌는지 확인해야 합니다. 배포 저장소를 push한 뒤에는 소스 저장소의 submodule pointer가 바뀌었는지도 확인해야 합니다. 자동화의 품질은 “명령을 많이 실행했는가”가 아니라 “각 단계의 결과를 검증했는가”로 갈립니다.

특히 Hermes/리오 같은 개인 에이전트 환경에서는 공개 글과 내부 운영정보의 경계가 중요합니다. 크론 지시, 로컬 작업 디렉터리, 채널 ID, 세션 키, 토큰, 계정 식별자는 작업에는 필요할 수 있지만 사용자에게 보이는 블로그 본문에는 필요 없습니다. 따라서 공개 산출물을 만들 때는 마지막에 “이 문장이 외부 독자에게 보여도 되는가”를 한 번 더 통과시켜야 합니다. 에이전트가 잘하는 일은 빠른 생성이지만, 신뢰를 만드는 일은 선택과 삭제입니다.

실전에서는 작은 체크리스트 하나로 충분히 좋아집니다. 작업 시작 전에는 입력 소스와 목표 산출물을 확인합니다. 쓰기 전에는 기존 파일명 충돌을 확인합니다. 커밋 전에는 변경 파일 목록을 확인합니다. 빌드 후에는 exit code와 생성 파일을 확인합니다. 최종 보고 전에는 공개 문구에 레거시 이름이나 내부 경로가 남지 않았는지 확인합니다. 이 다섯 단계만 있어도 자동화는 훨씬 안정적으로 변합니다.

결론적으로 Hermes/리오 자동화의 핵심은 “에이전트에게 더 많은 권한을 주는 것”이 아니라 **권한마다 다른 검문소를 붙이는 것**입니다. 읽기와 분석은 과감하게 자동화하고, 되돌릴 수 있는 내부 쓰기는 검증과 함께 실행하고, 외부 부작용은 명시된 범위 안에서만 진행해야 합니다. 이렇게 설계하면 에이전트는 단순한 스크립트 실행기가 아니라, 매일 반복되는 운영 업무를 안전하게 끝내는 작은 런타임이 됩니다.

실전 체크리스트:

- 읽기·내부 쓰기·외부 부작용을 먼저 구분한다.
- 커밋이나 배포 전에는 항상 변경 파일 목록을 확인한다.
- 공개 산출물에는 내부 식별자와 로컬 운영정보를 넣지 않는다.
- 빌드는 실행 여부가 아니라 결과 파일 생성까지 확인한다.
- 자동화가 실패하면 추측하지 말고 실패 단계와 다음 액션을 짧게 남긴다.
