오늘 AI/ML 트렌드에서 눈여겨볼 신호는 **코딩 에이전트의 컨텍스트 폭발**입니다. 지난주 GitHub의 AI credit pool 관리 기능과 NVIDIA의 NVLink 인프라 강화 발표가 있었는데, 그 배경에는 코드 실행 에이전트들이 생성하는 tool output의 양이 모델과 인프라의 설계 범위를 벗어나고 있다는 신호가 숨어 있습니다.

arXiv에 공개된 SWE-Pruner Pro 논문이 정확히 이 문제를 다룹니다. 코딩 에이전트가 실행 결과(에러 메시지, 로그, 빌드 출력)를 그대로 컨텍스트에 넣으면, 한 번의 코드 실행으로 수천 토큰이 소비됩니다. 컨텍스트 윈도우는 고정돼 있고, 토큰당 비용은 할인율이 떨어지므로, 에이전트는 자신이 읽어야 할 정보와 읽으면 안 되는 noise를 구분할 수 없습니다.

SWE-Pruner Pro의 해법은 단순하지만 효과적입니다. 에이전트가 tool output을 **동적으로 선별**하도록 학습시킵니다. 빌드 실패 메시지에서 실제 에러만 추출하고, 경고는 제거하고, 성공 로그는 한 줄로 축약합니다. 이를 통해 같은 컨텍스트 윈도우에서 5-10배 더 많은 작업 이력을 유지할 수 있게 됩니다.

하지만 이것이 단순한 LLM 최적화 문제는 아닙니다. 더 큰 그림을 보면:

1. **인프라 계층의 피드백**: GitHub의 credit pool 관리는 팀이 에이전트 실행 비용을 추적하기 시작했다는 뜻입니다. 무분별한 tool output 소비는 곧 팀의 예산 초과로 가시화됩니다.

2. **확장성의 벽**: NVIDIA가 NVLink 6세대를 통해 GPU 간 대역폭을 강조한 이유는, 프론티어 모델들이 더 많은 추론 단계와 tool 호출을 지원하려면 단순히 더 큰 배치보다는 더 빠른 네트워크가 필요하다는 걸 알기 때문입니다. 코딩 에이전트는 이 대역폭 소비의 최대 고객입니다.

3. **에이전트 설계의 변곡점**: SWE-Pruner Pro 같은 논문이 나오는 것은 "더 큰 모델, 더 많은 토큰" 식의 단순 확장이 한계에 다다랐다는 신호입니다. 이제는 **에이전트가 스스로 읽을 정보를 필터링**해야 합니다.

실무에서 이것이 의미하는 바:

- **코딩 에이전트 도입 시**: 모델 벤치마크와 함께 실제 코드 저장소에서 얼마나 많은 tool call을 생성하는지, 각 call이 몇 토큰을 소비하는지 측정하세요. 초기 파일럿에서 이를 무시하면 프로덕션 비용이 예상의 3-5배가 될 수 있습니다.

- **조직 수준에서**: GitHub의 credit pool 같은 기능을 도입했다면, 에이전트별 비용 추이를 월 단위로 검토하세요. 비용 증가 추세가 작업량 증가와 맞지 않으면, tool output 생성 패턴이 비효율적으로 변했을 가능성이 높습니다.

- **에이전트 아키텍처**: 도구를 호출할 때, "전체 결과를 컨텍스트에 넣기"와 "특정 필드만 추출해서 넣기" 사이의 선택을 에이전트가 학습할 수 있도록 설계하세요. 이를 강제하기보다는 피드백 신호(비용, 최종 코드 품질)를 통해 자동으로 선택하게 하는 게 효과적입니다.

결국 이 뉴스는 "코딩 에이전트도 리소스 제약에 빠진다"는 당연한 진실을 드러냅니다. 무한히 확장되는 컨텍스트나 모델이 없으므로, 에이전트는 **한정된 예산 안에서 정보를 재우선순위**해야 합니다. 좋은 에이전트는 더 길게 생각하는 게 아니라 더 현명하게 생각하는 에이전트입니다.
