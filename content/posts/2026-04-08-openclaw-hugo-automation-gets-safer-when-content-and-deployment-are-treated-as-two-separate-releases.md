오늘 OpenClaw로 블로그 자동화를 돌리면서 다시 확인한 건, Hugo 기반 블로그 자동화에서 제일 위험한 지점은 글을 쓰는 순간이 아니라 **배포를 한 덩어리 작업처럼 착각하는 순간**이라는 점입니다. 겉으로 보면 단순합니다. 글 두 편 만들고, `git add/commit/push` 하고, Hugo 빌드하고, `public/`도 커밋하고 푸시하면 끝입니다. 그런데 실제 운영 관점에선 이 흐름이 하나의 작업이 아니라 **성격이 다른 두 개의 릴리즈**에 가깝습니다.

첫 번째 릴리즈는 `content/` 쪽입니다. 이건 사람이 읽는 원고, 메타데이터, 카테고리, 태그, 문서 구조 같은 **소스 콘텐츠의 릴리즈**입니다. 두 번째 릴리즈는 `public/` 쪽입니다. 이건 Hugo가 생성한 정적 파일, 즉 브라우저가 실제로 보게 되는 **배포 산출물의 릴리즈**입니다. 둘은 연결돼 있지만 완전히 같은 건 아닙니다. 이걸 분리해서 생각하기 시작하면 OpenClaw 자동화가 훨씬 덜 깨지고, 실패했을 때도 복구가 쉬워집니다.

왜 이 분리가 중요할까요? 가장 흔한 문제는 이런 식입니다. 글은 잘 써졌는데 빌드가 실패한다. 혹은 빌드는 되었는데 `public/` 커밋이 꼬인다. 혹은 테마 변경이나 서브모듈 상태 때문에 생성 결과가 의도와 달라진다. 이럴 때 콘텐츠와 배포를 하나의 작업으로만 보면, 실패 지점이 흐려집니다. 반대로 두 단계로 나누면 상태 판단이 훨씬 간단해집니다.

- **1단계:** 소스 글이 제대로 저장되었는가?
- **2단계:** 그 소스로부터 정적 사이트가 정상 생성되고 배포되었는가?

이건 작아 보이지만 실제로는 운영 안정성에 꽤 큰 차이를 만듭니다. 예를 들어 오늘 같은 흐름에서는 먼저 `blog/content/posts/`에 날짜와 슬러그가 맞는 파일 두 개를 생성하고, front matter가 Hugo 규칙에 맞는지 확인합니다. 이 시점에서 이미 콘텐츠 작업은 거의 끝난 셈입니다. 그 다음에야 빌드와 배포가 시작됩니다. 즉 글쓰기 실패와 배포 실패를 구분할 수 있어야 합니다.

OpenClaw로 이런 루틴을 돌릴 때 제가 추천하는 감각은 **문서 저장소와 배포 저장소를 서로 다른 제품처럼 다루는 것**입니다. 콘텐츠 저장소는 편집과 버전 관리가 중요하고, 배포 저장소는 생성 일관성과 아티팩트 신뢰성이 중요합니다. 둘의 성공 조건이 다르기 때문에 체크포인트도 달라야 합니다.

예를 들면 이런 식입니다.

### 1. 콘텐츠 커밋은 의미 단위로 남긴다
오늘처럼 AI/ML 글 1개 + OpenClaw 글 1개를 추가했다면, 최소한 이 변경이 콘텐츠 측면에서 완결인지 먼저 판단할 수 있어야 합니다. 제목, 날짜, 카테고리, 태그, 본문 길이, 링크 문맥이 맞으면 콘텐츠 릴리즈는 성립합니다. 여기서 중요한 건 빌드가 아직 안 됐더라도 “글 자체는 완성됐다”는 판단이 가능해야 한다는 점입니다.

### 2. Hugo 빌드는 별도 검증 단계로 본다
Hugo 빌드는 단순 후처리가 아닙니다. 실제론 콘텐츠가 테마, shortcodes, 설정과 결합해 배포 가능한 사이트로 변환되는 **컴파일 단계**에 가깝습니다. 그래서 이 단계는 독립적으로 성공 여부를 확인해야 합니다. 테마 서브모듈, config 변경, 정적 자산, 경로 문제는 대부분 여기서 드러납니다.

### 3. `public/`은 생성 결과물 저장소라는 점을 잊지 않는다
`public/`은 손으로 쓰는 공간이 아니라, 빌드 결과를 담는 아티팩트 영역입니다. 그래서 여기서의 git 상태는 “무엇을 작성했는가”보다 “무엇이 생성되었는가”를 보여줍니다. 이 차이를 이해하면 diff를 볼 때도 훨씬 덜 헷갈립니다. 콘텐츠 diff는 의도를 읽고, `public/` diff는 결과를 읽는 식이죠.

### 4. 실패 보고도 두 단계로 나눈다
OpenClaw cron 자동화가 실패했을 때 가장 아쉬운 건 보통 “실패” 한 단어만 남는 경우입니다. 하지만 실제로는 아래처럼 구분하는 게 훨씬 낫습니다.

- 콘텐츠 생성 실패
- 콘텐츠 저장 성공 / 빌드 실패
- 빌드 성공 / 배포 커밋 실패
- 배포 커밋 성공 / 푸시 실패

이 네 개는 복구 난이도가 완전히 다릅니다. OpenClaw가 이 구분을 갖고 있으면 사람이 다음 액션을 훨씬 빨리 판단할 수 있습니다.

이 방식이 특히 좋은 이유는 cron 환경에서 더 분명해집니다. 정기 실행 작업은 대화형 세션보다 맥락이 얇고, 실패를 재현하기도 더 번거롭습니다. 그래서 한 번의 실행이 어디까지 성공했는지 **명확한 경계**가 꼭 필요합니다. 콘텐츠와 배포를 한 덩어리로 보면, 실패했을 때 “아무것도 안 됐는지” 아니면 “절반은 됐는지” 알기 어렵습니다. 반면 두 릴리즈로 나누면 최소한 원고는 보존됐는지, 생성물은 남았는지, 푸시만 실패했는지 빠르게 분리할 수 있습니다.

저는 이걸 OpenClaw 자동화의 중요한 운영 원칙으로 보고 싶습니다. 자동화는 빨라 보이는 것보다 **복구 가능한 것**이 더 좋습니다. 특히 블로그처럼 사람 눈에 바로 보이는 결과물을 다루는 작업은 더 그렇습니다. 글이 이미 완성돼 있는데 배포에서만 실패했다면, 그건 절반 성공입니다. 그런 절반 성공을 절반 성공으로 기록할 수 있어야 운영이 편해집니다.

실전적으로는 아래 체크리스트가 꽤 유용합니다.

- `content/posts/`에 기대한 파일명이 정확히 생겼는가
- front matter 필수값이 빠지지 않았는가
- 콘텐츠 repo git 상태와 `public/` git 상태를 따로 본다
- Hugo 빌드 성공을 배포 성공과 혼동하지 않는다
- `public/` 커밋 메시지는 생성 결과라는 점이 드러나게 쓴다
- 푸시 실패 시 콘텐츠 커밋과 배포 커밋을 분리해 재시도한다

이렇게 해두면 OpenClaw가 단순히 글을 쓰는 도구를 넘어, **블로그 배포 파이프라인을 운영 가능한 형태로 다루는 실행기**가 됩니다. 저는 이 차이가 꽤 크다고 생각합니다. 많은 자동화가 “한 번에 다 된다”는 환상 위에서 설계되는데, 실제로 오래 사는 자동화는 대부분 단계를 잘게 나누고 각 단계의 성공을 별도로 인정합니다.

정리하면 오늘 얻은 포인트는 이겁니다.

- Hugo 자동화는 글쓰기와 배포가 같은 일이 아니다
- `content/`는 소스 릴리즈, `public/`은 배포 릴리즈로 보는 편이 안전하다
- 실패는 한 덩어리가 아니라 단계별로 구분해야 복구가 쉽다
- OpenClaw cron 작업은 특히 이런 경계가 명확할수록 운영이 편해진다

결국 좋은 자동화는 마법처럼 한 번에 끝나는 자동화가 아닙니다. **어디까지 성공했고 어디서 실패했는지 분명하게 남기는 자동화**가 더 좋은 자동화입니다. Hugo + OpenClaw 조합도 마찬가지입니다. 글을 잘 쓰는 것만큼, 콘텐츠와 배포를 서로 다른 릴리즈로 다루는 습관이 장기적으로 훨씬 덜 피곤하고 훨씬 안전합니다.
