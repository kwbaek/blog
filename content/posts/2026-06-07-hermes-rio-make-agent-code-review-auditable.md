Hermes/리오로 코드 작성, 리뷰, 블로그 배포 같은 반복 작업을 자동화할 때 가장 위험한 착각은 “에이전트가 맞는 말을 했으니 충분하다”입니다. 실제 운영에서는 답변의 그럴듯함보다 **무엇을 읽었고, 어떤 변경을 했고, 어떤 검증을 통과했는지**가 더 중요합니다. 특히 코드리뷰 자동화는 PR에 코멘트를 다는 순간 팀의 merge 판단에 영향을 줍니다. 그래서 Hermes/리오 워크플로우에서는 코드리뷰를 단순한 자연어 의견이 아니라 감사 가능한 작업 단위로 설계하는 편이 좋습니다.

첫 번째 원칙은 입력 범위를 명확히 남기는 것입니다. 에이전트가 리뷰한 diff, 관련 파일, 테스트 결과, 최근 커밋, 사용자 지시를 구분해야 합니다. “전체 repo를 봤다”는 표현은 거의 항상 부정확합니다. 실제로는 특정 파일 몇 개, 특정 로그, 특정 테스트 출력만 봤을 가능성이 큽니다. 리뷰 결과에는 어떤 근거를 사용했는지 짧게라도 남겨야 합니다. 이렇게 하면 나중에 잘못된 판단이 나왔을 때 모델 탓만 하지 않고, 입력이 부족했는지, 검증이 빠졌는지, 규칙이 애매했는지 확인할 수 있습니다.

두 번째 원칙은 코멘트 등급을 나누는 것입니다. 모든 지적을 같은 무게로 쓰면 리뷰가 시끄러워집니다. Hermes/리오 자동화에서는 보통 `blocker`, `risk`, `suggestion`, `note`처럼 등급을 두는 편이 좋습니다. blocker는 테스트 실패, 명백한 보안 문제, 데이터 손실 가능성처럼 merge를 막아야 하는 항목입니다. risk는 조건부 위험입니다. suggestion은 코드 품질 개선이고, note는 참고 정보입니다. 이 등급이 있으면 사람은 에이전트 코멘트를 훨씬 빨리 읽을 수 있고, 자동화도 어디서 멈춰야 하는지 알 수 있습니다.

세 번째 원칙은 검증 명령과 결과를 분리해서 기록하는 것입니다. “테스트했습니다”라고 쓰는 것보다 어떤 명령을 실행했고, 결과가 어땠고, 실패했다면 어떤 대안을 시도했는지가 중요합니다. Hermes/리오가 빌드, 테스트, 정적 검사, 생성물 확인을 수행했다면 그 출력은 리뷰의 증거가 됩니다. 반대로 실행하지 못한 검증은 실행한 것처럼 쓰면 안 됩니다. 자동화의 신뢰는 성공을 크게 말하는 데서 생기지 않고, 못 한 일을 정확히 표시하는 데서 생깁니다.

간단한 템플릿은 이렇게 시작할 수 있습니다.

```markdown
## Agent Review Evidence
- Scope: reviewed diff/files/commands
- Blockers: merge를 막을 문제
- Risks: 조건부 위험과 필요한 확인
- Suggestions: 품질 개선 제안
- Verification: 실제 실행한 명령과 결과
- Escalation: 사람이 판단해야 할 항목
```

마지막으로, 리뷰 에이전트는 많이 말할수록 좋은 도구가 아닙니다. 좋은 Hermes/리오 리뷰는 조용해야 할 때 조용하고, 멈춰야 할 때 확실히 멈추며, 사람이 다시 확인할 수 있는 근거를 남깁니다. 에이전트 코드리뷰를 운영에 붙일 계획이라면 모델 선택보다 먼저 감사 로그, 코멘트 등급, 검증 기록, 사람 escalation 기준을 정하세요. 그래야 자동화가 생산성을 높이면서도 팀의 신뢰를 갉아먹지 않습니다.
