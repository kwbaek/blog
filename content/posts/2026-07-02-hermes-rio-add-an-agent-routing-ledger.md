오늘 GitHub Copilot의 자동 모델 선택과 AWS의 A2A gateway 흐름을 Hermes/리오 운영 관점으로 가져오면 실무 팁은 하나입니다. 자동화마다 **Agent Routing Ledger**를 남기는 것입니다. 에이전트가 하나일 때는 “어떤 모델로 답했는가” 정도만 봐도 충분해 보입니다. 하지만 크론, 블로그 생성, 리포트 작성, 코드 수정, 브라우저 확인처럼 작업 종류가 늘어나면 중요한 질문이 바뀝니다. 왜 이 도구를 썼는지, 왜 이 출처를 신뢰했는지, 왜 이 파일만 수정했는지, 어떤 단계에서 멈출 수 있었는지를 기록해야 합니다.

Agent Routing Ledger는 거창한 데이터베이스가 아니라 실행별 짧은 운영 장부입니다. Hermes/리오가 어떤 입력을 읽었고, 어떤 후보를 버렸고, 어떤 도구 호출을 했고, 어떤 검증을 통과했는지 남기는 습관에 가깝습니다. 예를 들어 블로그 포스팅 크론이라면 오늘의 AI/ML 리포트와 최신 트렌드 캐시를 확인했는지, 같은 날짜 글이 이미 있는지, 새 글이 500자 이상인지, Hugo 빌드가 통과했는지, 생성된 공개 HTML에 민감한 운영 세부값이 섞이지 않았는지를 기록해야 합니다.

체크리스트는 이렇게 시작할 수 있습니다.

```markdown
## Agent Routing Ledger
- Input: 이번 실행이 실제로 읽은 리포트, 캐시, 소스는 무엇인가
- Selection: 왜 이 주제를 골랐고 어떤 중복 후보를 제외했는가
- Permission: 이번 자동화가 허용받은 쓰기/커밋/배포 범위는 어디까지인가
- Routing: 어떤 도구를 어떤 순서로 호출했고 실패 시 대안은 무엇이었는가
- Cost/limit: 장시간 작업, 외부 호출, 반복 실행을 어디서 제한했는가
- Verification: 빌드, 테스트, 파일 존재, 민감정보 검사 중 무엇이 통과했는가
```

이 장부의 장점은 다음 실행을 더 똑똑하게 만든다는 점입니다. 문제가 생겼을 때 “모델이 이상한 답을 했다”로 끝나지 않고, 입력이 부족했는지, 라우팅 선택이 잘못됐는지, 검증이 빠졌는지 분리해서 볼 수 있습니다. 특히 사람 없이 도는 크론은 마지막 응답이 짧을수록 내부 근거가 중요합니다. 사용자에게는 성공/실패만 간결하게 알려주더라도, 에이전트 자신은 어떤 근거로 공개 배포까지 갔는지 재현할 수 있어야 합니다.

Hermes/리오를 개인 운영체제처럼 쓰려면 자동화를 많이 만드는 것만큼 자동화의 의사결정 흔적을 남기는 일이 중요합니다. 에이전트가 더 많은 도구와 모델을 선택하게 될수록 라우팅은 보이지 않는 운영 정책이 됩니다. Agent Routing Ledger는 그 보이지 않는 정책을 작게라도 문서화하는 방법입니다. 오늘부터는 “잘 실행됐다”보다 “왜 그렇게 실행됐는지 다시 설명할 수 있다”를 기준으로 에이전트 자동화를 관리하는 편이 좋습니다.
