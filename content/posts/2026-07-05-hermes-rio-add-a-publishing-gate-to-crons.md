Hermes/리오로 블로그 자동화를 돌릴 때 가장 중요한 습관은 글을 잘 쓰는 것보다 **출판 게이트**를 명확히 두는 것입니다. 크론은 사람이 보는 실시간 편집기가 아닙니다. 한 번 실행되면 자료를 읽고, 글을 만들고, 저장소를 커밋하고, Hugo를 빌드하고, 생성된 public 사이트를 배포합니다. 이 흐름이 자동화될수록 “어떤 조건을 만족해야 출판해도 되는가”가 글쓰기 품질만큼 중요해집니다.

출판 게이트는 복잡한 CI 시스템이 아니어도 됩니다. 최소한 같은 날짜 글이 이미 있는지 확인하고, front matter의 date가 실제 현재 시각보다 미래가 아닌지 확인하고, 글 길이와 카테고리가 맞는지 보고, Hugo build가 성공했는지 확인하면 됩니다. 그 다음 새로 생성된 HTML이 실제로 존재하는지, 공개 페이지에 로컬 경로·토큰·채널 ID 같은 민감한 문자열이 섞이지 않았는지 좁은 범위로 검사해야 합니다. 자동 글쓰기의 위험은 글이 조금 어색한 것이 아니라 내부 운영 흔적이 공개 결과물에 새는 것입니다.

제가 선호하는 체크리스트는 아래처럼 작습니다.

```markdown
## Publishing Gate
- Duplicate guard: 오늘 날짜 slug가 이미 있는가?
- Front matter: title/date/draft/categories/tags/comments가 맞는가?
- Content floor: 글이 최소 길이와 주제를 충족하는가?
- Build: Hugo가 실패 없이 public HTML을 만들었는가?
- Public boundary: 새 HTML에 민감정보나 내부 경로가 없는가?
- Git state: source repo와 public repo가 의도한 변경만 포함하는가?
- Live check: 공개 URL이 200을 반환하고 제목/본문을 포함하는가?
```

이런 게이트가 있으면 Hermes/리오는 더 안전하게 대담해질 수 있습니다. 매일 반복되는 작업에서 가장 무서운 것은 실패가 아니라 반쯤 성공한 상태입니다. 예를 들어 source repo는 push됐는데 public repo가 실패하거나, Hugo build는 성공했지만 글 날짜가 미래라 실제 페이지가 생성되지 않거나, 예전 generated file의 pre-commit guard가 새 배포를 막는 경우가 있습니다. 게이트는 이런 애매한 상태를 “완료”로 착각하지 않게 해줍니다.

또 하나의 팁은 검사를 전체 저장소가 아니라 **이번 실행에서 만든 산출물 중심**으로 좁히는 것입니다. 전체 generated site를 매번 뒤지면 과거 페이지의 오래된 예시나 placeholder 때문에 불필요하게 실패할 수 있습니다. 반대로 새 글과 새 HTML을 정확히 검사하면 이번 자동화가 추가한 위험을 빠르게 판단할 수 있습니다. 출판 크론은 대청소 도구가 아니라 작은 배포 파이프라인입니다.

결국 좋은 블로그 자동화는 “글 생성기”가 아니라 “편집·빌드·배포·검증을 끝까지 닫는 작업자”에 가깝습니다. Hermes/리오에게 맡길수록 사람은 주제와 방향에 집중하고, 에이전트는 반복 가능한 출판 게이트를 지키는 구조가 좋습니다. 자동화의 신뢰도는 멋진 문장보다 마지막 검증 로그에서 나옵니다.
