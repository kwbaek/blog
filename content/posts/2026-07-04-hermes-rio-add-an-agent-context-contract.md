오늘의 AI/ML 흐름을 Hermes/리오 운영 팁으로 바꾸면 핵심은 단순합니다. 자동화가 읽고 쓰는 컨텍스트에 대해 **에이전트 컨텍스트 계약서**를 붙여야 합니다. 에이전트가 한두 개의 파일만 읽을 때는 “필요한 자료를 알아서 찾아봐”도 괜찮아 보입니다. 하지만 크론, 리포트, 블로그, 배포, 캐시, MCP 도구가 연결되면 컨텍스트는 곧 운영 상태가 됩니다. 어떤 입력을 믿을 수 있는지, 어떤 상태를 갱신할 수 있는지, 무엇을 공개 문구에 절대 넣으면 안 되는지를 명시하지 않으면 자동화는 점점 불안정해집니다.

컨텍스트 계약서는 복잡한 문서가 아니어도 됩니다. 작업별로 입력 원천, 신뢰도, 갱신 권한, 실패 시 대체 경로, 검증 방법을 적어두면 충분합니다. 예를 들어 블로그 포스팅 크론이라면 입력은 최신 AI/ML 트렌드 캐시와 일간 리포트, 중복 기준은 최근 블로그 글, 쓰기 대상은 Hugo 글과 생성 사이트, 검증은 글 길이·빌드 성공·생성 HTML 확인·공개 문구 점검입니다. 이렇게 적어두면 다음 실행에서 “무엇을 근거로 글을 썼는지”와 “어디까지 자동으로 해도 되는지”가 분리됩니다.

제가 추천하는 최소 양식은 아래와 같습니다.

```markdown
## Agent Context Contract
- Purpose: 이 자동화가 만드는 결과물
- Trusted inputs: 우선 읽을 리포트, 캐시, 문서
- Fallback inputs: 주 입력이 없을 때 허용되는 대체 자료
- Writable state: 갱신해도 되는 파일, 캐시, 아티팩트
- Read-only state: 참고만 하고 수정하지 않을 자료
- Stop conditions: 추측하면 안 되고 멈춰야 하는 상황
- Verification: 빌드, 링크, JSON 검증, 공개 문구 점검 등 완료 기준
- Public boundary: 사용자에게 보여도 되는 정보와 숨겨야 하는 내부 정보
```

이 계약서가 중요한 이유는 에이전트의 실수가 대부분 “모델이 몰라서”가 아니라 “어떤 상태를 믿어야 하는지 애매해서” 생기기 때문입니다. 오래된 캐시를 최신으로 착각하거나, 내부 점검 문구를 공개 글에 섞거나, 대체 소스를 써도 되는 상황과 실패로 보고해야 하는 상황을 구분하지 못하면 자동화 품질이 흔들립니다. 반대로 계약서가 있으면 Hermes/리오는 도구를 많이 써도 더 침착하게 움직일 수 있습니다.

특히 주기적으로 실행되는 크론에는 컨텍스트 계약서가 작은 안전벨트가 됩니다. 매번 사람이 지켜보지 않아도 입력 우선순위와 검증 기준이 고정되기 때문입니다. 에이전트를 잘 쓰는 방법은 더 많은 권한을 주는 것이 아니라, 읽을 수 있는 상태와 쓸 수 있는 상태를 선명하게 나누는 것입니다. 자동화가 늘어날수록 “컨텍스트를 잘 관리하는 능력”이 곧 운영 실력입니다.
