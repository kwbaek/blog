오늘 AI/ML 신호(뉴욕 주정부의 데이터센터 승인 보류, Reflection의 10억 달러 컴퓨트 계약)가 던지는 실무 교훈은 하나입니다. **당신의 자동화가 어떤 컴퓨트·사업자·지역에 걸려 있는지 모르면, 그 뒤의 공급 제약이 조용히 당신을 망가뜨린다.** Hermes(리오)로 매일 크론을 돌리는 사람에게 이건 남의 얘기가 아닙니다. 스캐너, 리포트, 블로그 포스팅 크론은 전부 특정 모델·API·지역 인프라 위에서 돕니다.

그래서 오늘 제안하는 습관은 **크론마다 '인프라 의존성'을 한 블록으로 선언**해두는 것입니다. 거창한 도구가 아니라, 지시문 상단에 5줄짜리 명세를 박아두는 일입니다.

## 왜 필요한가

크론의 실패는 보통 두 종류입니다. 하나는 로직 오류(코드가 틀림). 다른 하나는 **의존성 붕괴** — 모델이 사라지거나, 사업자 지역이 제약되거나, API 쿼터가 바뀌는 것. 로직 오류는 로그에 흔적을 남기지만, 의존성 붕괴는 "왜 갑자기 느려졌지", "왜 이 리전만 실패하지" 같은 **원인 불명의 지연**으로 나타납니다. 무엇에 의존하는지 명시돼 있지 않으면 디버깅이 추측 게임이 됩니다.

## 크론 지시문에 넣을 인프라 의존성 블록

```markdown
## 인프라 의존성 (Infra Dependency)
- 주 모델 / 폴백: <primary> → <fallback>
- 외부 API: <검색·RSS·메일·메신저 등 나열>
- 암묵적 지역 의존: <데이터센터/리전 가정, 없으면 "없음">
- 컴퓨트 민감도: 높음 / 보통 / 낮음  (지연·중단 시 영향)
- 저하 시 동작: 폴백 모델 사용 / 캐시로 진행 / [SILENT] 종료
```

리오에게는 이렇게 지시하면 됩니다.

> "이 크론을 실행하기 전에 인프라 의존성 블록을 확인해. 주 모델이나 외부 API가 응답하지 않으면 명시된 '저하 시 동작'을 따르고, 추측으로 대체 결과를 지어내지 마. 폴백을 썼거나 캐시로 진행했으면 최종 응답 맨 끝 한 줄로 어떤 저하가 있었는지 기록해."

## 오늘 신호와 어떻게 연결되나

- **데이터센터 허가 리스크**가 커지면, 특정 지역 인프라에 걸린 크론은 지연·비용 증가에 노출됩니다. '암묵적 지역 의존'을 적어두면, 뉴스에서 그 지역 제약이 나올 때 어떤 크론을 먼저 점검할지 즉시 알 수 있습니다.
- **효율화 흐름(ACRouter식 작업별 라우팅)**과도 맞물립니다. 각 크론의 '컴퓨트 민감도'를 적어두면, 무거운 모델이 꼭 필요한 작업과 경량 모델로 충분한 작업을 구분해 **불필요한 고비용 호출을 줄일 수** 있습니다.
- **폴백을 실제로 검증**하게 됩니다. 선언만 하고 끝내지 말고, 주 모델을 강제로 막은 상태에서 크론을 한 번 돌려 저하 시 동작이 지시대로 작동하는지 확인하세요.

## 오늘 5분 액션

1. 매일 도는 크론 중 가장 중요한 것 하나를 고른다.
2. 위 인프라 의존성 블록을 지시문 상단에 추가한다.
3. 리오에게 폴백/캐시 저하 시나리오를 한 번 시뮬레이션시키고, 결과 한 줄 기록이 제대로 남는지 본다.

자동화의 신뢰성은 "얼마나 똑똑한 모델을 쓰느냐"가 아니라 **"그 모델이 사라졌을 때 무엇을 하도록 미리 정해뒀느냐"**에서 갈립니다. 인프라 의존성 선언은 그 대답을 크론 안에 박아두는 가장 싼 보험입니다.
