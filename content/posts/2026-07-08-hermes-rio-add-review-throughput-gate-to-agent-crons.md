Hermes/리오로 블로그 작성, 리포트 생성, 코드 수정, 배포 보조 같은 작업을 자동화하다 보면 “작업을 끝냈다”는 말의 기준이 중요해집니다. 파일을 썼다고 끝난 것이 아닙니다. 테스트를 돌리고, 빌드하고, 생성 산출물을 확인하고, 공개될 문장에 내부 운영 흔적이 섞이지 않았는지 살펴봐야 합니다. 그래서 저는 개발 자동화 크론에도 **review throughput gate**를 붙이는 것이 좋다고 봅니다.

GitHub이 Copilot usage metrics API에 review cycle과 time-to-review 지표를 추가한 것은 개인 자동화에도 시사점이 큽니다. AI가 코드를 빠르게 만들수록 병목은 “작성”이 아니라 “검토”로 이동합니다. 에이전트가 PR 초안을 만들고, 테스트를 실행하고, 설명을 작성하더라도 사람이 리뷰할 때 맥락이 부족하면 오히려 반복이 늘어납니다. Hermes/리오 자동화도 마찬가지입니다. 결과물을 많이 만드는 것보다, 검토자가 빠르게 판단할 수 있는 증거를 함께 남기는 것이 더 중요합니다.

제가 추천하는 review throughput gate는 간단합니다.

```markdown
| 항목 | 확인 질문 |
|---|---|
| 변경 범위 | 이번 작업이 어떤 파일과 산출물만 바꿨는가? |
| 검증 증거 | 테스트, 빌드, 렌더링 확인 중 무엇을 실제로 실행했는가? |
| 리뷰 부담 | 사람이 봐야 할 핵심 diff와 위험 지점은 무엇인가? |
| 반복 가능성 | 같은 크론이 내일 다시 돌 때 중복·충돌을 피할 기준이 있는가? |
| 공개 안전성 | 최종 공개물에 내부 경로, 식별자, 민감 문자열이 섞이지 않았는가? |
```

이 표는 거창한 승인 절차가 아닙니다. 자동화가 끝날 때마다 “리뷰가 쉬운 상태인가?”를 묻는 작은 문입니다. 예를 들어 블로그 게시 크론이라면 Markdown 파일 작성만으로는 부족합니다. Hugo build가 성공했는지, 새 HTML이 실제로 생성됐는지, 새 공개 페이지에 불필요한 운영 표현이 없는지 확인해야 합니다. 코드 수정 크론이라면 lint와 테스트 결과, 실패한 경우의 원인, 사람이 볼 핵심 변경점을 남겨야 합니다.

특히 반복 크론에서는 review cycle을 줄이는 설계가 필요합니다. 매번 비슷한 실수를 고치느라 사람이 다시 지시해야 한다면 자동화의 생산성은 허상입니다. Hermes/리오는 최근 캐시, 이전 리포트, 스킬 노트, git 상태를 읽을 수 있으므로 “어제와 같은 주제를 반복하지 않기”, “이미 존재하는 파일을 덮어쓰지 않기”, “빌드 전에 미래 날짜 글을 만들지 않기” 같은 체크를 자동으로 수행해야 합니다. 이런 사전 확인이 리뷰 반복을 줄입니다.

또 하나 중요한 점은 요약 방식입니다. 자동화가 끝난 뒤에는 긴 로그를 그대로 보여주는 것보다, 사람이 판단할 수 있는 짧은 완료 증거가 낫습니다. 무엇을 만들었는지, 어떤 검증이 통과했는지, 남은 unrelated 변경이 있는지 정도면 충분합니다. 내부 작업 경로와 운영 식별자는 공개 보고에 넣지 않는 편이 안전합니다. 좋은 자동화는 내부적으로는 꼼꼼하고, 외부로는 필요한 사실만 말합니다.

정리하면 Hermes/리오 자동화의 품질은 실행 속도보다 리뷰 처리량에서 갈립니다. 에이전트가 빠르게 결과물을 만드는 시대에는 “얼마나 많이 만들었나”보다 “사람이 얼마나 적은 반복으로 승인할 수 있나”가 더 중요한 지표입니다. 크론마다 review throughput gate를 붙이면 자동화는 단순 생산 도구가 아니라 신뢰할 수 있는 운영 동료에 가까워집니다.
