OpenClaw로 블로그 자동화를 굴릴 때 제가 점점 더 강하게 느끼는 원칙이 하나 있습니다. **콘텐츠 작성과 사이트 배포를 한 덩어리로 취급하지 않는 편이 훨씬 안전하다**는 점입니다. 얼핏 보면 글을 만들고 바로 빌드해서 퍼블릭 저장소까지 밀어 넣는 것이 가장 효율적으로 보입니다. 하지만 실제 운영에서는 이 둘을 분리해 생각해야 사고가 줄어듭니다. 글 작성은 편집 작업이고, 배포는 릴리즈 작업이기 때문입니다.

특히 OpenClaw의 cron 기반 워크플로에서는 이 차이가 더 중요합니다. cron은 정해진 시각에 실행되는 만큼, 매번 같은 품질로 반복 가능해야 합니다. 그런데 자동 글쓰기와 정적 사이트 배포는 실패 양상이 다릅니다. 글 작성 단계에서는 제목, front matter, 카테고리, 태그, 본문 구조처럼 콘텐츠 품질이 중요합니다. 반면 배포 단계에서는 Hugo 빌드 성공 여부, 생성 결과 차이, `public/` 반영 상태, Git 커밋 순서처럼 릴리즈 무결성이 중요합니다. 이 둘을 한 번에 뭉개면 어디서 문제가 났는지 찾기 어려워지고, 실패 복구도 훨씬 불편해집니다.

그래서 OpenClaw 운영에서는 보통 다음 순서가 가장 깔끔합니다.

1. `content/posts/`에 새 글을 만든다.
2. 블로그 레포에서 먼저 커밋한다.
3. Hugo 빌드로 산출물을 생성한다.
4. `public/`을 별도 커밋으로 배포한다.

이 흐름의 장점은 단순하지만 큽니다. 첫째, 글 자체의 변경 이력과 실제 배포 결과를 분리해 볼 수 있습니다. 둘째, 빌드가 깨졌을 때 콘텐츠 커밋은 남기고 배포만 보류할 수 있습니다. 셋째, 나중에 문제가 생겨도 “글이 이상했는지”와 “배포가 이상했는지”를 빠르게 구분할 수 있습니다. 자동화에서 가장 무서운 건 실패 그 자체보다 **실패 원인이 흐려지는 것**인데, 두 단계 릴리즈는 바로 그 문제를 줄여 줍니다.

OpenClaw와 잘 맞는 이유도 있습니다. 이 도구는 instruction file, cron, git, 빌드 같은 작업을 차례대로 실행하기에 좋고, 각 단계를 명시적으로 남기는 습관과도 잘 맞습니다. 즉 단순히 일을 대신 해 주는 것을 넘어서, **반복 가능한 운영 절차를 에이전트가 지키게 만들기 쉬운 구조**입니다. 사람이 직접 할 때는 귀찮아서 생략하기 쉬운 커밋 분리나 빌드 확인 같은 것도, 에이전트에게는 오히려 명확한 체크포인트가 됩니다.

저는 이런 차이가 장기적으로 꽤 크다고 봅니다. 자동화는 빠른 것보다 믿을 만한 것이 더 중요합니다. 오늘은 잘 됐는데 내일은 조용히 틀어지는 자동화는 결국 사람을 더 피곤하게 만듭니다. 반대로 작성과 배포를 분리하고, 각 단계를 독립된 책임으로 다루면 OpenClaw 워크플로는 훨씬 덜 불안해집니다. 블로그 자동화를 꾸준히 돌릴 생각이라면, 제일 먼저 거창한 프롬프트를 손보기보다 **콘텐츠 릴리즈와 사이트 릴리즈를 나누는 구조**부터 잡는 편을 추천합니다. 그게 결과적으로 가장 boring하고, 그래서 가장 좋은 자동화입니다.
