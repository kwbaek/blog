OpenClaw를 오래 운영해보면, 자동화 성공 여부는 모델 성능보다 **운영 구조**에서 갈리는 경우가 많습니다. 최근 제가 가장 효과를 본 패턴은 “크론 잡에 긴 프롬프트를 직접 넣지 않고, `cron-instructions/` 폴더의 지시문 파일을 읽게 만드는 방식”입니다. 이 구조를 쓰면 디버깅이 쉬워지고, 잡 수정도 빠르며, 실패 원인 추적도 명확해집니다.

핵심 아이디어는 단순합니다. 크론 잡 본문은 최소화하고, 실제 작업 규칙은 파일로 분리합니다. 예를 들어 블로그 자동화라면 (1) 트렌드 채널 읽기, (2) 카테고리별 포스트 생성, (3) git push, (4) 빌드/배포 같은 단계를 지시문 파일에 명시해 둡니다. 이렇게 하면 규칙 변경 시 크론 자체를 건드리지 않고 파일만 수정하면 됩니다.

운영 관점에서 장점은 세 가지입니다. 첫째, **재현성**: 같은 파일을 읽으면 같은 기준으로 동작합니다. 둘째, **가시성**: 실패 시 “어느 단계에서 깨졌는지”를 파일 순서대로 확인할 수 있습니다. 셋째, **보안/위생**: 민감정보(토큰, 세션키, 로컬 절대경로)를 출력하지 않도록 지시문에 명시하면, 로그 노출 위험을 줄일 수 있습니다.

실무에서 특히 중요한 디테일은 경로 규칙입니다. 절대경로에 의존하면 실행 환경이 바뀔 때 바로 깨집니다. 반대로 `cron-instructions/`, `scripts/`, `config/`처럼 상대경로 우선 원칙을 지키면 호스트/샌드박스 차이에 더 탄력적으로 대응할 수 있습니다. 또한 “파일이 없으면 1줄 실패 보고 후 종료” 같은 단순한 예외 규칙을 넣어두면, 장애 상황에서도 운영자가 빠르게 복구할 수 있습니다.

정리하면, OpenClaw 자동화의 품질은 ‘얼마나 똑똑하게 쓰게 하느냐’보다 ‘얼마나 안정적으로 반복되게 설계하느냐’에 달려 있습니다. 지시문 분리 + 경로 표준화 + 실패 시 명확한 종료 규칙. 이 세 가지만 지켜도 크론 운영 피로도가 크게 줄어듭니다.

## 추천 체크리스트
- 크론 본문은 짧게, 규칙은 `cron-instructions/*.md`로 분리
- 상대경로 우선 (`cron-instructions/`, `scripts/`, `config/`)
- 실패 보고 포맷 고정(원인 1줄 + 다음 액션 1개)
- 민감정보 비출력 원칙을 지시문에 명시
- 배포 전 `git status`와 빌드 결과를 항상 확인
